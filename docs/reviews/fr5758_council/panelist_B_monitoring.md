# Panelist B — FR-58 (Live monitoring cadence): Soundness & Buildability

**Charter:** Adversarial review of decoupling the activity-refresh cadence from the stall threshold and making the table print regularly. Is the design sound and buildable?

**Verdict: REVISE-REQUIRED** — the core idea (a finer, configurable keepalive/activity-refresh interval decoupled from the stall threshold) is sound, cheap, and a clean fit for the existing `_Keepalive` clock seam. But the spec as written has three defects that must be fixed before build: (1) it conflates the *heartbeat-emit* cadence with the *activity-rescan* cadence and never states they are the same event (they are, and should stay so — but the spec must say it and the design must not accidentally over-write the heartbeat file); (2) it leaves a pre-existing wiring gap unaddressed — the adapter today ignores the plan's stall/grace values entirely, so a naive `--keepalive-seconds` add inherits a config surface that is already half-broken; (3) the "regular visible table" half is almost entirely guidance, not engine, and the spec must say so honestly rather than implying a buildable change.

---

## Finding 1 — Decoupling is sound, but the two cadences are NOT two events; the spec must say one event, finer

**The double-duty concern is real but resolves cleanly.** Today `keepalive_interval_secs(launch_grace, stall) = max(1, min(launch_grace, stall/3))` (heartbeat.py:393-398) is the single interval that does both jobs: (a) keep the heartbeat young enough that FR-8 stall detection (`heartbeat age > threshold ⇒ STALLED`, REQUIREMENTS.md:236) never false-fires, and (b) drive the activity re-scan + IN_PROGRESS emit in `_Keepalive.maybe_emit` (heartbeat.py:432-450).

The clean design is **one timer, finer, that drives both the emit and the re-scan — exactly as today, just a shorter interval.** The activity re-scan and the heartbeat emit are *already the same event* in `maybe_emit` (the `feed()` call at heartbeat.py:441 and the `append_line()` at :445 are in the same `if due` block). FR-58 should NOT split them into two timers. Splitting buys nothing and costs: a second scheduler, a second clock-seam test surface, and the risk that the re-scan fires without an emit (so the operator's ACTIVITY label updates in the matcher's memory but never lands in the heartbeat file the engine renders). Keep them coupled. The spec's phrase "drives IN_PROGRESS keepalives **and** re-scans the activity patterns" (REQUIREMENTS.md:367) is correct in intent but must be made explicit that this is ONE due-check at the fine interval, not two intervals.

**What decoupling actually means here:** decouple the *keepalive interval* from the *stall threshold's arithmetic* (`stall/3`), NOT decouple the re-scan from the emit. The stall threshold stays coarse (45 min default) and continues to govern FR-8 STALLED marking in the engine; the keepalive interval becomes an independent ~30–60 s knob. The spec's framing ("independent *of* the stall threshold") is right; it just needs the one-event clarification so a builder doesn't introduce a second scheduler.

**Does a 30–60 s keepalive risk masking a real stall?** No — and this is the load-bearing soundness check. Stall detection lives in the *engine* (FR-8), which compares heartbeat *age* against `stall_threshold_minutes`. A finer keepalive makes the heartbeat *younger more often*, which can only *delay* a STALLED verdict if the worker is genuinely hung — BUT the keepalive is **timer-driven, not output-driven** (heartbeat.py:432-450, the `due()` check is a pure function of `now/last_emit/interval` and does not read the child). A hung-but-alive wrapped child still emits keepalives at any interval, fine or coarse, so the keepalive *already* doesn't catch a hung-but-alive child — that's by design (FR-40: "a silent job never false-STALLs"). What catches a truly dead worker is the **dead-PID reap** (A-5 backstop, referenced at heartbeat.py:603-617 and REQUIREMENTS.md:411), not heartbeat age. So shortening the keepalive does NOT mask any stall that coarse keepalives would have caught. **No regression on stall semantics.** (One honest caveat the spec should note: a finer keepalive means the heartbeat file is never older than ~60 s, so the FR-8 age-based STALLED path becomes effectively unreachable for adapter jobs while the wrap/tail process is alive — but that was already true at the 10-min keepalive vs 45-min stall, just with more margin. FR-58 doesn't introduce this; it widens an existing gap. Worth one sentence of disclosure.)

**NFR-13 (tick cost) — fine.** NFR-13 says "idle ticks are cheap by design so over-polling is always safe" (REQUIREMENTS.md:384). A keepalive emit is one `append_line` (one `O_APPEND` write of a sub-200-byte JSON line, heartbeat.py:93-104). At 30 s × a multi-hour job that's a few hundred writes — negligible. The matcher `feed()` is bounded by the FR-56 per-scan 256 KiB ceiling (heartbeat.py:357) regardless of how often it runs. **No NFR-13 violation.** But see the next sub-point.

**The FR-56 byte-ceiling-per-scan interaction is the one genuine soundness wrinkle, and the spec misses it.** The 256 KiB ceiling is *per scan* (heartbeat.py:353-358), and `feed()` only scans `new_lines()` (the incremental delta since the last call, heartbeat.py:441 / `_LogTail.new_lines` :547-560). So the *total* bytes scanned over a run is the file size, fed in deltas, independent of scan frequency — EXCEPT: the ceiling is a per-scan budget, so a chatty tool that emits >256 KiB *between two fine scans* now hits the ceiling and **drops** the lines past the budget for that scan, retaining the older label (heartbeat.py:357-358). Counterintuitively, scanning MORE often makes each delta SMALLER, so finer keepalives make the ceiling *less* likely to bite, not more — the spec's worry ("the activity re-scan running too often") is backwards. The real (small) risk is the opposite: at a *coarse* interval a big delta could blow the budget; the fine interval mitigates that. Net: finer is strictly better for activity fidelity. The spec should correct the framing ("re-scan running too often" is not a risk; the per-scan-ceiling-vs-delta-size relationship is the actual mechanism) so the builder doesn't add a needless throttle.

## Finding 2 — It fixes the symptom, with two named gaps

**Confirmed root cause.** For the ~16 s stub job: `keepalive_interval_secs(10*60, 45*60) = min(600, 900) = 600 s` (10 min). The wrap loop (`proc.wait(timeout=interval)`, heartbeat.py:509) blocks for 600 s; the 16 s child exits first; `maybe_emit` is *never reached* (the `except TimeoutExpired` at :511 never fires). So the heartbeat stream is exactly STARTING (heartbeat.py:476) + terminal (heartbeat.py:518) — the FR-56 matcher's `feed()` is never called, ACTIVITY stays at whatever STARTING set (`"wrap: <cmd>"`). **The diagnosis in the brief is exactly right.**

**Does a finer interval fix it?** For a job *longer than the fine interval*: yes. With `--keepalive-seconds 30`, `proc.wait(timeout=30)` times out at 30 s, `maybe_emit(now)` fires, `feed()` scans the captured output, `label()` returns the matched line, the IN_PROGRESS lands, the engine renders it. ACTIVITY moves. **The fix is real and mechanical.**

**Gap 2a — a job shorter than the fine interval still shows nothing.** A 16 s job under a 30 s keepalive STILL emits only STARTING + terminal — same failure, smaller window. This is unavoidable for the keepalive path (you can't emit a mid-work ping before the first interval elapses) and the spec must state it honestly rather than implying "the ACTIVITY column then moves" universally (REQUIREMENTS.md:367). Two honest options the spec should pick between: (i) accept it and document that sub-interval jobs are too short to show intermediate activity (defensible — there's nothing meaningful to show in 16 s); or (ii) add a **first-scan-at-start** — call `activity.feed()` + emit one IN_PROGRESS immediately after STARTING, before entering the wait loop, so even a sub-interval job surfaces at least one activity line. Option (ii) is ~4 lines in `_cmd_wrap`/`_cmd_tail` and genuinely closes the demonstrated symptom (the 16 s stub would then show a real ACTIVITY line). **I'd require option (ii)** — it's cheap and it's the literal scenario that birthed FR-58. The spec currently specifies neither; that's the buildability gap.

**Gap 2b — the test that "verified" FR-56 used grace 0.** The FR-56 EndToEnd tests "drive the REAL wrap + tail subprocess (grace 0)" (REQUIREMENTS.md:425). With grace 0, `keepalive_interval_secs` floors to 1 s, so the keepalive fires and FR-56 looked verified. The default-grace path (10 min interval) was never exercised end-to-end — which is *why* the real stub test exposed STARTING-stuck ACTIVITY. The FR-58 acceptance test must drive the **default** keepalive path (a short job under a realistic ~30 s keepalive, asserting ≥1 IN_PROGRESS with a matched label lands), not grace 0 — otherwise FR-58 ships with the same blind spot that hid the bug.

## Finding 3 — The "regular visible table" half is guidance, not engine; the spec is mostly honest but must be sharper

**The engine already does its part.** FR-5 emits `status_table` in the per-tick JSON (REQUIREMENTS.md:233), every tick, unconditionally. The operator didn't see it because the rung-1 agent ran the ticker inside one collapsed bash block — a *behavioral* failure of how the agent surfaced the engine's existing output, not a missing engine feature. The spec correctly identifies this ("not captured/suppressed, not buried in one long ticker bash block", REQUIREMENTS.md:368).

**So this half is ~100% SKILL/runbook, ~0% engine** — and the spec must say so plainly. As written it's *mostly* honest (it attributes the change to "orchestrator + SKILL") but it blurs the line by phrasing the requirement as "the ticker prints a clearly-delimited table each tick" (REQUIREMENTS.md:368), which reads like a `ticker.py` change. Decide and state: is there ANY engine/ticker delta (e.g. a per-tick delimiter banner around `status_table`, or a `ticker.py` flag that prints the table to its own stdout line instead of inside the JSON blob)? If yes, scope it as a buildable item with a test. If no — and I believe no is correct, the engine already emits it — then say "FR-58 part 2 is a SKILL/runbook requirement with NO engine change; the test is the FR-50/positioning-honesty doc-guard pattern (`test_positioning_honesty.py`), not a code test." The honesty bar this project holds elsewhere (the §9 VERIFIED-vs-DESIGNED discipline, NFR-12) demands FR-58 not imply buildable code where it's prose guidance. **Require: an explicit one-line engine-vs-guidance split in the FR-58 text.**

One concrete engine-adjacent option worth offering (not requiring): a tiny `ticker.py` enhancement to print the `status_table` as a delimited block to stdout each `--once` tick (separate from the machine-readable JSON), so the SKILL guidance has a clean primitive to surface rather than asking the agent to parse-and-reprint JSON. That IS buildable and testable. If the council wants the table-visibility to have *any* mechanical backstop rather than pure runbook discipline, that's where it goes. But it's optional; the runbook-only path is defensible given the engine already emits the data.

## Finding 4 — Config surface: sound shape, but it inherits a pre-existing wiring break the spec must fix in the same change

**`--keepalive-seconds` + a plan field is the right shape** — it mirrors the existing `--launch-grace-minutes`/`--stall-threshold-minutes` argparse pattern (heartbeat.py:729-732, 758-759) and the `_TOPLEVEL_PASSTHROUGH` plan-field pattern (jobs.py:41-43). Clean.

**But there is a latent break the spec walks right past, and FR-58 must address it.** `_adapter_worker_cmd` (tick.py:1081-1117) synthesizes `--task-id`, `--heartbeat-path`, `--activity-regex`, and the tail markers — but it **never synthesizes `--launch-grace-minutes` or `--stall-threshold-minutes`**. Those args default to `None` in argparse (heartbeat.py:729-731, 758-759), and `_grace_stall_secs` (heartbeat.py:251-261) then falls back to the hardcoded `_DEFAULT_LAUNCH_GRACE_MIN=10` / `_DEFAULT_STALL_THRESHOLD_MIN=45` (heartbeat.py:247-248). **Consequence: the plan's top-level `stall_threshold_minutes`/`launch_grace_minutes` are read by the engine for FR-8 stall detection but are NEVER passed to the wrapped adapter's keepalive math.** So today an operator who sets `stall_threshold_minutes: 12` in the plan still gets a 10-min adapter keepalive (from the hardcoded 45/3=15 → min(600,900)=600 s) — the adapter and the engine disagree about the same plan.

This matters for FR-58 because: if you add `--keepalive-seconds` *only* as a heartbeat.py arg without synthesizing it in `_adapter_worker_cmd`, you reproduce the exact same dead-knob bug — the plan field would be inert for adapter jobs (the only jobs that use the keepalive at all). **Require: `_adapter_worker_cmd` synthesizes `--keepalive-seconds` from the plan field (and, while you're there, fix the pre-existing miss by also synthesizing grace/stall — or at minimum document that they're intentionally adapter-defaulted).** Without this, the config surface is decorative.

**Clean interval resolution.** Add an explicit keepalive override that takes precedence over the derived floor:

```
def keepalive_interval_secs(launch_grace_secs, stall_secs, override_secs=None):
    if override_secs is not None:
        return max(1.0, float(override_secs))     # explicit knob wins, 1 s floor
    return max(1.0, min(float(launch_grace_secs), float(stall_secs) / 3.0))
```

- **Default value:** ~45 s (middle of the 30–60 band). Sub-minute so the operator perceives motion; not so fine that a multi-day job writes hundreds of thousands of lines. Make it an explicit named constant (`_DEFAULT_KEEPALIVE_SECS = 45`) alongside the grace/stall defaults (heartbeat.py:247-248) so it's discoverable and test-pinnable.
- **Interaction with launch_grace:** the override must still land the first keepalive *within* launch_grace or FR-8 LAUNCH-FAIL could fire before the first ping. At a 45 s keepalive vs a 10 min default grace, fine — the first ping lands at 45 s, well inside 600 s. But guard it: if `override_secs > launch_grace_secs`, the first keepalive would miss the grace window. Either (a) `--check` rejects `keepalive_seconds > launch_grace_minutes*60` (cleanest — fail loud at validation), or (b) clamp `min(override, launch_grace)` for the *first* interval only. **I'd require the `--check` rejection** — it's consistent with the project's fail-at-`--check` posture (FR-42) and avoids silent clamping surprises.
- **Interaction with stall:** none needed — the override is decoupled from `stall/3` by design (that's the whole point). The engine's FR-8 stall threshold stays independent. Just confirm in a test that a fine keepalive + a coarse stall threshold coexist (the keepalive keeps the heartbeat young; the stall threshold governs the engine's age check; they don't collide).
- **Postel posture:** match the existing `_grace_stall_secs` discipline (heartbeat.py:251-261) — an explicit non-positive value floors to 1 s (it documents exactly this for grace/stall), not silently replaced by the default. Reuse that pattern so the `is not None` / floor-at-1 behavior is consistent across all three knobs.

## The clean design I'd require before build

1. **One finer timer, not two.** The activity re-scan and the IN_PROGRESS emit stay a single due-check in `_Keepalive.maybe_emit`; FR-58 only shortens the interval. Spec must state "one event at the fine interval," not "two cadences."
2. **`keepalive_interval_secs(grace, stall, override=None)`** — explicit override wins, floors to 1 s, falls back to the existing `min(grace, stall/3)` derivation. `_DEFAULT_KEEPALIVE_SECS = 45`.
3. **First-scan-at-start** (the ~4-line addition in `_cmd_wrap`/`_cmd_tail`): emit one activity-scanned IN_PROGRESS immediately after STARTING, so a sub-interval job (the 16 s stub that birthed FR-58) shows ≥1 real ACTIVITY line. Without this, FR-58 does not fix its own motivating scenario.
4. **Synthesize `--keepalive-seconds` in `_adapter_worker_cmd`** (tick.py:1097-1116) from a plan field, alongside fixing/documenting the pre-existing grace/stall non-synthesis. A plan knob that doesn't reach the adapter is inert.
5. **`--check` rejects `keepalive_seconds > launch_grace_minutes*60`** (fail-loud per FR-42), so the first ping can't miss the grace window.
6. **Part 2 (visible table) labeled honestly** as SKILL/runbook-only (the engine already emits `status_table` every tick, FR-5) — OR scope the optional `ticker.py` delimited-table-print as an explicit, tested engine item. No blurring.
7. **Acceptance test drives the DEFAULT keepalive path** (a short real wrap/tail job at ~30–45 s keepalive, asserting ≥1 IN_PROGRESS with a matched activity label), NOT grace 0 — closing the blind spot that hid the original bug.

All seven are stdlib-only, clock-seam-testable (the `ARUNNER_NOW` / `_now()` seam at heartbeat.py:278-287 already exists for exactly this), cross-platform (NFR-1/3), and small. The idea is sound and buildable; the spec just needs these corrections so a builder doesn't reproduce the grace-0 blind spot, the inert-plan-knob bug, or a second-scheduler over-design.
